# Bug 反馈标准化处理流程

## 触发条件
当用户说"发现 Bug"、"有问题"、"报错了"、"不正常"时自动激活。

## 处理流程

### 第一步：接收原始反馈 → 追问补全

收到 Bug 反馈后，自动检查以下字段是否完整：

| 字段 | 是否必填 | 追问话术 |
|------|:---:|------|
| 📌 版本号 | ✅ | "在哪个版本上发现的？（如 v1.2.0）" |
| 💻 复现环境 | ✅ | "用的什么浏览器/设备？测试环境还是线上？" |
| 👤 账号信息 | ✅ | "用哪个账号操作的？" |
| ⚙️ 前置条件 | ✅ | "操作前做了什么？（登录/选了xx/配置了xx）" |
| 🔄 复现步骤 | ✅ | "能按 1→2→3 列出具体操作步骤吗？" |
| ❌ 实际结果 | ✅ | "实际上发生了什么？有截图或录屏吗？" |
| ✅ 期望结果 | ✅ | "你期望的正确结果是什么？" |
| 📸 截图/录屏 | 建议 | "方便截个图吗？" |
| 📋 日志/报错 | 建议 | "控制台有红色报错吗？F12→Console 看看" |
| 📊 影响范围 | 建议 | "这个问题影响了哪些功能/用户？" |

### 第二步：严重度 & 优先级判定

```
             核心功能受损？
            /          \
          是            否
          │             │
     阻断主流程？      次要功能？
    /        \        /      \
  是         否      是       否
  │          │       │        │
 P0         P1      P2       P3
阻断上线   严重    一般     轻微
```

**自动判定规则：**
- 支付/登录/核心业务 → 直接 P0
- 数据丢失/安全漏洞 → 直接 P0
- 有 workaround 的功能异常 → P2
- UI 错位/文案错误 → P3

### 第三步：输出标准 Bug 描述

```markdown
## Bug #{编号}：{一句话标题}

| 属性 | 值 |
|------|-----|
| **严重程度** | 🚨 P0 / P1 / P2 / P3 |
| **优先级** | P0/P1/P2/P3 |
| **阻断上线** | 是 / 否 |
| **版本** | v{version} |
| **环境** | 测试/预发/生产 |
| **账号** | {account} |
| **复现率** | 100% / 偶发 / 特定条件 |
| **影响范围** | {模块/用户范围} |

### 前置条件
{描述}

### 复现步骤
1. {步骤1}
2. {步骤2}
3. {步骤3}

### 实际结果
{实际发生了什么}

### 期望结果
{应该发生什么}

### 附件
- 📸 截图: {link}
- 📋 日志: {link}

### 关联
- 需求: {PRD link}
- 测试用例: {TC link}
```

### 第四步：创建事项

> 将此 Bug 创建为 GitHub Issue（使用 🐛 Bug 反馈模板），分配研发负责人，设置 `bug:P{0-3}` 标签，关联原需求 PRD。
