## 一、模块定位

Goal 推进模块是 OPS 系统的**「中枢神经」**——它不是独立阶段，而是横切六阶段、贯穿全程的**推进引擎**。

```
                    ┌──────────────────────────────────┐
                    │        Goal 推进模块              │
                    │                                  │
                    │  需求评估 → 任务拆解 → 增量实施    │
                    │       │           │         │     │
                    │       ▼           ▼         ▼     │
                    │  每周报告 → 检查点 → 结果验证      │
                    └──────────┬───────────────────────┘
                               │ 贯穿六阶段
          ┌────────────────────┼────────────────────┐
          ▼                    ▼                    ▼
    ┌──────────┐        ┌──────────┐        ┌──────────┐
    │ 阶段一    │        │ 阶段二    │   ...  │ 阶段六    │
    │ 测试准备  │        │ Bug反馈   │        │ 质量复盘  │
    └──────────┘        └──────────┘        └──────────┘
```

**核心原则：不替代六阶段流程，而是给每个阶段注入推进纪律。**

## 二、命名规则（沿用 OPS 资料库规范）

| 产物 | 命名格式 | 示例 |
|------|---------|------|
| 需求评估报告 | `GOAL_REQ_{需求编号}_{日期}.md` | `GOAL_REQ_OPS-001_20260615.md` |
| 任务拆解清单 | `GOAL_TASK_{版本号}_{日期}.md` | `GOAL_TASK_v1.0.0_20260615.md` |
| 每周推进报告 | `GOAL_WEEKLY_{年份}W{周号}.md` | `GOAL_WEEKLY_2026W25.md` |
| 检查点报告 | `GOAL_CHECKPOINT_{Phase}_{日期}.md` | `GOAL_CHECKPOINT_Phase1_20260622.md` |
| 结果验证报告 | `GOAL_RESULT_{版本号}.md` | `GOAL_RESULT_v1.0.0.md` |

**存储路径：** `/规范文档/Goal推进/`

## 三、全生命周期闭环

```
需求进来
    │
    ▼
┌─────────────────────────────────────────────────────────┐
│  ① 需求评估 (阶段一之前)                                    │
│  ─────────────────                                       │
│  输入: PRD / 用户故事 / 需求文档                            │
│  动作: interview-me → idea-refine → 输出评估报告            │
│  产出: GOAL_REQ_{需求编号}_{日期}.md                        │
│  判据: 需求是否足够清晰、可验证、有验收标准                    │
└──────────────────────┬──────────────────────────────────┘
                       │ 评估通过
                       ▼
┌─────────────────────────────────────────────────────────┐
│  ② 任务拆解 (阶段一)                                       │
│  ─────────────────                                       │
│  输入: 评估通过的需求                                       │
│  动作: planning-and-task-breakdown → 依赖图 → 垂直切片       │
│  产出: GOAL_TASK_{版本号}_{日期}.md                         │
│  判据: 每个任务有验收标准 + 验证步骤 + 预估量级                │
└──────────────────────┬──────────────────────────────────┘
                       │ 拆解完成
                       ▼
┌─────────────────────────────────────────────────────────┐
│  ③ 增量实施 (阶段二→四)                                    │
│  ─────────────────                                       │
│  输入: 任务清单                                            │
│  动作: incremental-implementation → 薄切片 → 每个增量验证     │
│  判据: 每次提交 ≤100行, 测试通过, Feature Flag 控制           │
│  每个增量完成 → 更新 GitHub Issue 状态                       │
└──────────────────────┬──────────────────────────────────┘
                       │
          ┌────────────┼────────────┐
          ▼            ▼            ▼
     每周一自动    每 Phase 结束   版本结束时
     生成周报      生成检查点      生成结果报告
```

## 四、核心机制

### 4.1 每周推进报告（自动生成）

**触发：** 每周一 09:00（automatic）或手动触发

**数据源：** GitHub Issues 状态 + 检查点报告

**报告模板：**

```markdown

|------|-----|
| **报告周** | 2026-W{NN} ({月}/{日} → {月}/{日}) |
| **当前活跃 Phase** | {Phase 名称} |
| **总任务数** | {N} |
| **本周完成** | {X} |
| **累计完成** | {Y}/{N} ({百分比}%) |
```

### 4.2 检查点（Phase 级别）

每个 Phase 结束自动触发，不可跳过：

| 检查点 | 位置 | 判据 |
|:---:|------|------|
| CP-1 | Phase 1 结束 | 完整 Bug 生命周期走通 + 资料库目录就绪 |
| CP-2 | Phase 2 结束 | 测试用例生成准确率 ≥ 90% + Bug 字段补齐率 ≥ 95% |
| CP-3 | Phase 3 结束 | CI 门禁成功阻断 ≥ 1 次 + 安全扫描无高危 |
| CP-4 | Phase 4 结束 | 巡检成功捕获 ≥ 1 个回归 + 回归清单自动生成可用 |
| CP-5 | Phase 5 结束 | Grafana 看板可查看全链路 + 告警正确触发 |
| CP-6 | Phase 6 结束 | 复盘报告自动生成 + ADR 自动入库 |

**检查不通过 → 不得进入下一 Phase。**

### 4.3 结果验证

版本结束时，Goal 推进模块自动对比：**计划 vs 实际**。

## 五、与 OPS 六阶段的嵌入关系

| OPS 阶段 | Goal 推进动作 | 触发时机 | 产出 |
|----------|-------------|------|------|
| **阶段一之前** | 需求评估 (`GOAL_REQ_`) | 新需求进入 | 评估报告：可执行/需澄清/不可行 |
| **阶段一** | 任务拆解 (`GOAL_TASK_`) | 需求评估通过 | 依赖图 + 垂直切片 + 验收标准 |
| **阶段一→六** | 每周报告 (`GOAL_WEEKLY_`) | 每周一 09:00 | 进度/阻塞/趋势 |
| **每 Phase 结束** | 检查点 (`GOAL_CHECKPOINT_`) | Phase 最后一个 Task 完成 | 通过/不通过 + 问题清单 |
| **阶段六结束** | 结果验证 (`GOAL_RESULT_`) | 版本验收通过 | 计划 vs 实际 + 偏差分析 |

## 六、稳定性保障机制

```
┌──────────────────────────────────────────────────────┐
│                  三重保险                              │
│                                                      │
│  ① 检查点阻断                                          │
│     每个 Phase 结束必须通过检查点，不通过 → 不进入       │
│     下一 Phase。防止问题滚雪球。                          │
│                                                      │
│  ② 每周自动报告                                        │
│     每周一份进度报告，任何人（包括管理层）都能看到         │
│     真实推进状态。没有地方藏问题。                         │
│                                                      │
│  ③ 偏差早发现                                          │
│     如果实际速率持续低于计划速率 2 周以上，                │
│     自动升级 → 通知 TL → 分析原因 → 调整计划。            │
└──────────────────────────────────────────────────────┘
```

## 七、稳定性指标

| 指标 | 当前基准 | 目标 | 衡量方式 |
|------|:---:|:---:|------|
| 需求评估通过率 | — | ≥ 80% | 评估通过数 / 总需求数 |
| 任务按期完成率 | — | ≥ 85% | 按期完成 / 总任务 |
| 检查点一次通过率 | — | ≥ 70% | 一次通过 / 总检查点 |
| 阻塞平均解除时间 | — | ≤ 2 天 | 阻塞创建→解除 的均值 |
| 计划偏差率 | — | ≤ 15% | (实际工期 - 计划工期) / 计划工期 |
| 周报准时率 | — | 100% | 准时出报周数 / 总周数 |
